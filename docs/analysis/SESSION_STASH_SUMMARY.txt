PROPOSED DESIGN:
================

SESSION IDENTIFICATION:
  Header: X-Session-ID (client-provided)
  Fallback: Auto-generated from IP + timestamp
  Purpose: Correlate multiple execute() calls to same session

STORAGE ARCHITECTURE:
  SessionManager (global, in server.py)
    ├─ SessionStash per session_id
    │   ├─ _data: Dict[key] = value
    │   ├─ _ttl_map: Dict[key] = expiration_time
    │   └─ Methods: get(), put(), has(), delete(), clear(), keys()
    └─ cleanup_expired(): Remove stale sessions

KEY DECISIONS:
==============

1. WHY HEADER-BASED SESSION ID?
   + Clients can explicitly identify sessions
   + Survives SSE streaming model (GET + POST)
   + Backward compatible (optional)
   + Correlates multiple execute() calls
   - Requires client cooperation (not automatic)

2. WHY IN-MEMORY ONLY?
   + Simple, no external dependencies (no Redis)
   + Fast, no I/O overhead
   + Suitable for single-process gateway
   - Not suitable for distributed deployments

3. WHY PER-KEY TTL?
   + Fine-grained control over cache expiration
   + Search results (1h) vs schema (4h) can have different TTLs
   + Matches Forgemax API pattern

4. WHY SUBPROCESS DOESN'T PERSIST DIRECTLY?
   + Subprocess is sandboxed, isolated execution
   + Server.py coordinates stash state
   + Stash is server-side, not embedded in subprocess
   + Cleaner separation of concerns

CHALLENGES & MITIGATION:
========================

Challenge: Session ID reliability
  Risk: Clients don't set X-Session-ID, get auto-generated
  Mitigation: Warn in logs, clients should use UUIDs

Challenge: Cross-namespace access
  Risk: One namespace shouldn't read another's stash
  Solution: Can enforce namespaced keys (future enhancement)

Challenge: Memory exhaustion
  Risk: Sessions accumulate unbounded data
  Mitigation: Per-key TTL, per-session TTL, cleanup task

Challenge: Concurrent access
  Risk: Multiple requests in same session race on stash
  Mitigation: asyncio.Lock in SessionManager

Challenge: Subprocess updates lost
  Risk: Stash put/delete in subprocess not reflected server-side
  Solution: Subprocess returns stash delta, server applies updates

ADVANTAGES:
===========

+ Minimal changes: Mostly additive
+ No external dependencies
+ Backward compatible (optional header)
+ Namespace-aware (respects existing access control)
+ TTL-based cleanup (prevents memory leaks)
+ Client-controlled (explicit session IDs)
+ Observable (stash keys in response)
+ Thread-safe (asyncio.Lock)

DISADVANTAGES:
==============

- In-memory only (not suitable for multi-process)
- Requires client support (X-Session-ID header)
- Subprocess can't directly mutate stash (needs server coordination)
- No persistence across restarts
